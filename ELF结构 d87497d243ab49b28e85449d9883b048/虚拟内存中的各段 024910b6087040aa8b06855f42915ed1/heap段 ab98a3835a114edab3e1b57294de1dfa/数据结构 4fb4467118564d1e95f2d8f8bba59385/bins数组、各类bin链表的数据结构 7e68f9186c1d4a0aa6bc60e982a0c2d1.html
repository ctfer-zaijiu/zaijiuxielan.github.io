bins数组、各类bin链表的数据结构

malloc一个chunk的顺序：
  1. 对于小块的分配，先查找fast bin，再查找small bin，最后查找unsorted bin。
  1. 对于大块的分配，先查找large bin，再查找unsorted bin。
  1. 对于超级大块的分配，直接mmap

一、整体介绍

一个arena对应地由一个malloc_state结构体维护，该结构体中有域分别指向以下这两个数组：

1、fastbinsY数组：大小为[10]，用来存储7个fastbins。其中[7][8][9]是冗余设计。

2、bin数组：大小为[64]，用来存储unsorted bin、 small bins、large bins。

其中：

各个下标对应的bin类型如下：
  • bin[0]是null，作为冗余设计
  • 1 个 unsorted bin，存储在bin[1]
  • 62 个 small bins，存储在bin[2]至bin[63]
  • 63 个 large bins，存储在bin[64]至bin[126]
一个数组元素的数据类型如下：
struct malloc_chunk {
    INTERNAL_SIZE_T prev_size; /* 前一个块的大小（如果已释放） */
    INTERNAL_SIZE_T size;      /* 包含开销的总大小（以字节为单位） */
    struct malloc_chunk* fd;   /* 双向链表（仅在已释放时使用） */
    struct malloc_chunk* bk;   /* 只用于大块：指向下一个更大的块 */
    struct malloc_chunk* fd_nextsize; /* 双向链表（仅在已释放时使用） */
    struct malloc_chunk* bk_nextsize; /* 只用于大块：指向下一个更大的块 */
};

二、具体结构

（一）fastbin

1-思想

它的设计目的是：设计得像高速缓存一样，从而尽可能地实现加快其访问速度。

2-相关机制

为了提高速度，使用了以下3个机制：

  • 其日常被free时不主动执行“日常整理碎片块机制”，而是集中在某些特定的时刻用一个专门的malloc_consolidate函数来做碎片块整理合并。以减少其做碎片块整理合并的次数，节约时间。
    注意，但是它的物理相邻下一个chunk或上一个chunk如果执行了“日常整理碎片块机制”，还是有可能导致它被合并。
  • fastbin是单向链表，这样在表头插入删除节点会比双向链表更快，同时使用原子操作进行link和unlink。
  • 不维护它的任何相关标志位。

其具体的加入取出行为如下：

LIFO：新加入的块link入链表头，取出时从链表头取出

3-数据结构

宏定义：

当一个块被free时，会判断它的size是否≤global_max_fast宏。如果符合，则将其放入fastbin中。

可以通过mallopt函数修改global_max_fast宏定义的值：
mallopt(1,0); //将global_max_fast修改为0，这样chunk被free后就不会进入fastchunk

块大小：

fastbinY[i]对应的大小为“(2i+4) * sizeof(void)”

其中：

sizeof(void)的大小在x32中为4，在x64中为8。这个fastbin大小的最大值由宏定义描述 。

此处所说的块大小是包含块头的大小。

也就是说：

大小范围是0x10~0x40B——32位。0x20~0x80B——64位。一共七种大小。

相邻bin存放的大小相差0x8B——32位。0x10B——64位。

数组-链表结构：

fastbinY数组的大小是[10]，是一个指针数组。其中[0]~[6]各自指向一条fastbin链表，[7][8][9]作为冗余保留。

其中，数组元素和每个块的fd指向链表下一个块，链表尾的那个块的fd=NULL。

需注意的是，每一个块的下一块的P标志位必==1（这是因为它从free到重新malloc的过程中都不动它的P标志位）

（二）small bin

1-数据结构

块大小：

大小范围是：0x10-0x1f0B（x64位：0x20-0x3f0）

相邻bin存放的大小相差：0x8B（x64位：0x10）

有一个快速的对应公式，chunk_size = 2*SIZE_SZ *index

bin[2]~bin[63]结构细节：

是 62 个 smallbins，对应 62 个循环双向链表。每一个链表内部的各个节点是等大的。

3-large bin

【数据结构】

块大小：

大小范围是：0x200B~无穷

x64下是0x400B~无穷

bin[64]~bin[126]结构细节：

bin[64]~bin[126]对应的是63个双向循环链表，占用，每一个链表按从大到小顺序存储大小在某一个大小区间的块，具体来说：

  • 前 32 个 bins：从 0x200B 开始每次+0x40B
  • 接下来的 16 个 bins：从0xA00B开始每次+0x200B
  • 接下来的 8 个 bins：从0x2A00B开始每次+0x1000B
  • 接下来的 4 个 bins：从0xAA00B开始每次+0x8000B
  • 接下来的 2 个 bins：从0x2AA00B开始每次+0x40000B
  • 接下来的1个bin（最后一个）：从0xAAA00B开始到无穷大
例如，第一个bin就是存储在[0x200,0x240]这个区间内大小的freechunk
  • 链表&块

    如图，fd和bk还是像smallbin那样去形成双向链表，而fd_nextsize、fd_nextsize：

    则只有当其前后有大小不同的chunk时才会被启用，否则其他时间里都会被设置为0：

    这是一种“跳表”的设计思想

    链表初始化时，每条链表不会初始化有一个空头节点，链表的首个节点就是链表的第一个有效节点。

    如果链表只有一个chunk，则其fd域指向他自己

【操作】

遵循best fit原则，即：取出满足大小的最小chunk。

如果malloc函数想在large bin的某条链表中取出一个chunk给程序使用，但该链表中的所有free chunk都比我要取出的chunk小，但它后面的large bin链表里面有比它大的free chunk：

malloc函数会遍历从下一条链表开始的所有 large bin 链表，直到找到一个 free chunk 大小超过所需大小的链表，从该链表中分离出一个 free chunk，并将其分裂为所需大小的 chunk 和一个剩余大小的 free chunk。

3-unsorted bin

【定义】

大小超过global_max_fast的 chunk 被释放时会加入到这里，多次free就会多次把free chunk加入其中，一直等到要malloc了，就才会遍历判断切割unsofted chunk并把它们丢到small bin或large bin。

这个设计其实就是为了实现large bin和small bin就必须要有一个切割过程，而这个切割过程我们就可以根据延迟绑定的思想设计unsorted bin。

其进出顺序是FIFO，即进去时会被插入到链表头部，出来时从链表尾部取出

【数据结构】

他是一个双向循环链表，bin[1]指向该链表头，使用fd和bk成员作为指针，表中的元素是不做大小排序的

【使用操作】

【free】

当程序free一个chunk，会先判断该free chunk大小是否大于global_max_fast，如果大于， 就将其加入unsofted bin。

【malloc】

malloc会按照传参大小，使其向8B对齐，然后判断对齐后的大小是否小于global_max_fast，如果大于就表示要到unsorted bin中找free chunk。

分配器从最后一个unsorted chunk开始，遍历整个unsofted bin链表，并对链表中的每个节点做如下检查：

  1. 如果unsorted chunk满足以下几个条件，它就会被切割为一块满足申请大小的chunk和另一块剩下的chunk，前者返回给程序，后者重新回到unsorted bin。
    • 申请大小属于small bin范围
    • unosrted bin中只有该chunk
    • 切割之后的大小依然可以作为一个chunk
  1. 否则，从unsorted bin中删除该unsorted chunk，并对其处理后事：
    • 若unsorted chunk恰好和申请大小相同，则直接返回这个chunk
    • 若unsorted chunk属于small bin范围，插入到相应small bin
    • 若unsorted chunk属于large bin范围，插入到相应large bin：判断对应large bin为空，若是，直接插入unsorted chunk，其fd_nextsize与bk_nextsize指向自身。若否，则按大小降序插入对应large bin。

直到找到满足要求的unsorted chunk。

【如何获得unsorted chunk】

总共有以下三种来源：

1、当一个较大的（在bin中的）chunk（由于malloc）被分割成两半之后，如果剩下的部分大于MINSIZE，就会被放到unsortedbin中。

2、释放一个不属于fastbin的chunk，并且该chunk不和top_chunk紧邻时，该chunk会首先被放到unsortedbin中。

3、当进行malloc_consolidate时，如果不是和top_chunk近邻的话，可能会把合并后的chunk放到unsortedbin中。

malloc_consolidate的意思是堆中的碎片整理

三、glibc2.27新增：tcache机制

【定义】

glibc 2.26开始引入了 tcache

它也是不合并的类似于fast bin的单向链表结构，相当于一个范围扩大版的fastbin（因为fastbin的范围只有0x10-0x40（64位0x20-0x80）太小了）

【结构】

这是一个结构体，结构体中有两个数组指针，指向一个counts数组和一个entries数组。

其中，entries数组的每个元素是一个指向一条空闲chunk链表的指针，counts数组每个元素对应entries数组相同下标哪个元素，表示那条链表上面有多少个节点（最多7个）

这两个数组都是64个元素，范围是从0x10开始每次+0x8（x64是从0x20开始每次+0x10）

  • 结构体定义代码如下
    typedef struct tcache_entry{
  struct tcache_entry *next;//这个 next 和 fd 是重叠在同一个位置上的
} tcache_entry;


typedef struct tcache_perthread_struct{
  char counts[TCACHE_MAX_BINS];
  tcache_entry *entries[TCACHE_MAX_BINS]; // 宏定义TCACHE_MAX_BINS = 64
} tcache_perthread_struct;

【基本操作】

【tcache_put】

定义：

用于把一个 chunk 放到 指定的 tcache->entries 里面去，其中，传参tc_idx是他要放到的entries数组元素的下标，通过 csize2tidx (nb)（nb是chunk的大小）得到。

这个机制的设计目的类似于fast bin，但是它的优先级比fastbin高

操作流程：

它首先把 chunk+2*SIZE_SZ（就是除去 header部分） 强制转换成 tcache_entry *类型，然后插入到 tcache->entries[tc_idx]的首部，最后把 tcache->counts[tc_idx]加 1，表示新增了一个 chunk到 该 表项。

  • 函数详细代码如下：
    static __always_inline voidtcache_put (mchunkptr chunk, size_t tc_idx){
  tcache_entry *e = (tcache_entry *) chunk2mem (chunk);
  assert (tc_idx < TCACHE_MAX_BINS);
  e->next = tcache->entries[tc_idx];
  tcache->entries[tc_idx] = e;  // 增加到链表头部
  ++(tcache->counts[tc_idx]);  // 记录当前 bin 的 chunk数
}

【tcache_get】

定义：

根据 tc_idx 取出 tcache->entries[tc_idx] 的第一个chunk ，返回其指针

  • 函数详细代码如下：
    static __always_inline void *tcache_get (size_t tc_idx){
  tcache_entry *e = tcache->entries[tc_idx];
  assert (tc_idx < TCACHE_MAX_BINS);
  assert (tcache->entries[tc_idx] > 0);
  tcache->entries[tc_idx] = e->next;
  --(tcache->counts[tc_idx]);
  return (void *) e;
}

【malloc】

根据需要申请的块大小，转换成对应的索引tc_idx
