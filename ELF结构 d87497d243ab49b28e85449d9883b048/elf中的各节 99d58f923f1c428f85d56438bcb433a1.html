总览：

目标文件相关的各个节主要可以按照目的分为五类：、

1.一类是为了方便静态链接器进行链接

2.一类是为了方便调试，在真正打包成程序发行的时候往往会去掉这部分

3.一类是为了方便动态链接

4.一类是为了程序运行需要

5.一类是为了装载

  • elf头-一生万物

    【定义】

    elf文件头表有节头表的索引和节头表字符串表的索引，结合二者可以得到一个完整的节头表->即：当ld访问文件头表，就可以通过文件头表访问到节头表，再通过节头表访问到其他所有节

    （正是一生二二生三三生万物！）

    【结构-代码】

    typedef struct{ 
  	unsigned char e_ident[16]; //见下面解析

	Elf32_Half e_machine;//elf文件的cpu平台属性，如：其值为EM_386的话表示只能在Intel x86的cpu上运行，更多对应关系如下：
	Elf32_Word	e_version;//还是elf文件的主版本号，一般为1呗
	Elf32_Addr	e_entry;//***程序入口（开始执行程序指令处）虚拟地址，对.o文件来说，该值为0

    【关于e_ident】

    其中，魔术字符的历史渊源：

    因为过去的操作系统运行可执行文件是将ip指针直接从文件的第一个字节开始执行，所以当时人们就在文件的最开始放一条jmp指令来跳过文件头的数据内容到达可执行文件的真正入口。这条jump指令的对应二进制正好是0x01、0x07，这就是.out文件必定有的开头。

    后来的操作系统已经不会直接拿着ip指针上来就执行了，但为了兼容.out文件，就开发了魔术字符的设定，并将0x01、0x07作为了.out文件的魔术字符。

  • .eh_frame节-异常处理相关、挂钩子

    【定义】

    是一种特殊的节（section），用于存储异常处理（exception handling）相关的信息

    【相关机制】

    在C++中，异常处理机制允许程序在运行时捕获和处理异常。当异常发生时，程序会沿着调用栈（即函数调用的序列）向上查找，直到找到匹配的异常处理程序。

    【存储规则】

    .eh_frame节存储了与这种栈展开过程相关的信息，包括每个函数的调用帧（frame）以及异常处理相关的信息。

    该节包含了一系列的帧描述条目（Frame Description Entries，FDEs）和地址范围描述条目（Address Range Description Entries，ARLEs）。其中：

    FDEs：描述了函数调用帧的信息，例如函数起始地址、代码长度等。

    ARLEs：描述了代码段中与异常处理相关的地址范围，例如函数的代码地址范围。

    【存在条件】

    1、在调试信息中包含函数调用栈，例如gcc用到了“-g”选项

    2、代码中使用到了异常处理机制（如C++中的try、catch语句）

    【应用】

    在PWN-patch里面，可以用这个节来放数据，因为这个段中的信息不会影响程序的正常运行，所以不会被checkdown！
