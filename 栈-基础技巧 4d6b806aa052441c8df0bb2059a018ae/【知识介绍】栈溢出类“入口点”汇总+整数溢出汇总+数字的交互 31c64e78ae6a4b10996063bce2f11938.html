一、“入口点”汇总

1-输入相关函数

syscall直接调用read：

会一直中断监听读取某个文件(这里是stdin)，直到读到：

1、指定的长度

2、到达文件末尾（原理是os维护文件描述符的数据结构也维护了文件大小，如果read到文件末尾，os会给read提示；stdin是没有文件末尾的）

注意：它在遇到”\0”的时候不会终止读取，遇到“\n”也不会终止读取

read函数：

它的本质就是通过syscall直接调用read，所以同上即可~

注意：

1、它在遇到”\0”的时候不会终止读取，遇到“\n”会被截断(不知道为什么，可能是管道或者socket被“\n”终止中断的特性？)

2、它不会停下来等待用户输入，而是会直接读stdin缓冲区

gets函数：

会一直阻塞监听读取stdin，直到遇到“换行符”

其原理是循环调用read(0,buff,1)读1字节，然后判断这个字节是不是”\n”如果是就结束

注意：它在遇到”\0”的时候不会终止读取

scanf函数：

会一直中断监听读取stdin，直到遇到“空格、制表符或换行符”

注意：它在遇到”\0”的时候不会终止读取

2-strcpy函数

【定义】

如果把strcpy的目的字符串指针指向一个位于栈上的字符串，并且对源字符串不做溢出检查&不使用strncpy限制字符拷贝个数的话，如下：

char buffer[512];
strcpy(buffer, input);

就可以实现“通过让源字符串的长度溢出，从而使栈上的目的字符串溢出”的效果

【示例】

环境：x32，假设fuck函数的retaddr到局部变量buf之间的偏移是0x60

对于以下程序：

viod str1to2(char* str1){
	char str2[512];
  strcpy(str2, str1);
  puts(str2);
}

int main(){
	char str1[1024]={0};
	read(1,str1);
	str1to2(str1);
	return 0;
}

可以通过将input构造为：

payload = b"a"*512+b"a"*offset+b"一些栈上要填充的东西"+p64(要跳转到的地址)

实现在buf处溢出写入一些东西并ret2任意地址

【漏洞扩大】

    观察fuck函数在调用_strcpy之前，会将buf地址放在eax，不管栈地址怎么随机化，eax必定指向buf地址

    .text:0804841A                 lea     eax, [ebp+buffer]
.text:08048420                 push    eax             ; dest

    这样也就相当于ret到了buf处执行我们的shellcode

三、交互过程中，数字的处理

1、传给程序数字的时候，要考虑在两种情况下分别传给他不同的类型：

如果接收端有像scanf(”%i”,buf)这样的把控制台输入字符串转数字格式底层字节存储的情况，就可以直接传参字符串形式的数字，即：

str(number2send).encode()

如果接收端的是像scanf(”%s”,buf)这样的没有把控制台输入字符串转数字格式底层字节存储的情况，就要手动做转换，即：

p32：将数字转成16进制4B的小端存储的字节流

p64：将数字转成16进制8B的小端存储的字节流

2、接收程序传参的时候，要考虑在两种情况下分别进行不同的转换：

如果它是泄露方式是像puts()这样直接字符串输出，那我们就要把它按照小端序读出数字，即：

u32：将一个4B字节流当作一个16进制4B的小端存储的数字，然后翻译成数字

u64：将一个8B字节流当作一个16进制8B的小端存储的数字，然后翻译成数字

如果它是泄露方式是像printf(”%i”)这样当成一个数字或者像printf(”%p”)这样的，即：

atoi(number2recv)
